Gambar 1. Use Case Transaksi Absensi

Deskripsi Use Case Transaksi - Tarik Data Full Finger

ID Use Case UCT1
Nama Use Case Tarik Data Full Finger
Tujuan Memungkinkan HRD untuk melakukan pengambilan data absensi secara lengkap dari mesin fingerprint (FTM) dan menyimpannya ke dalam database sistem untuk keperluan pemrosesan kehadiran dan laporan.
Aktor Primer HRD
Aktor Sekunder a. FTM (Fingerspot/Mesin Fingerprint)
b. HRIS (Sistem Informasi SDM Perusahaan)
c. Database Sistem Absensi
Deskripsi Singkat Use case ini menjelaskan proses saat HRD melakukan penarikan data absensi secara full/lengkap dari mesin fingerprint. Sistem akan terhubung ke FTM, mengambil semua data absensi dalam periode tertentu, melakukan validasi dengan data karyawan dari HRIS, dan menyimpannya ke database sistem.
Precondition
  1. HRD telah login ke sistem
  2. Koneksi ke mesin fingerprint (FTM) tersedia
  3. Data mapping PIN-NIK sudah dikonfigurasi
  4. Data karyawan dari HRIS tersedia untuk validasi
Postcondition
  1. Data absensi dari fingerprint tersimpan di database sistem
  2. HRD menerima notifikasi atau log hasil proses tarik data
  3. Data siap untuk diproses di modul verifikasi atau report
  4. Log aktivitas pull data tercatat di sistem
Skenario Utama
  1. HRD login ke sistem absensi
  2. HRD membuka menu "Transaksi > Tarik Data Finger" atau "Pull Finger"
  3. Sistem menampilkan form dengan parameter:
    • Pilihan mesin fingerprint (jika ada multiple lokasi/mesin)
    • Rentang tanggal (dari tanggal - sampai tanggal)
    • Opsi: Tarik semua data atau hanya data baru
  4. HRD memilih mesin fingerprint yang akan di-pull
  5. HRD menentukan periode tanggal data yang akan diambil
  6. HRD klik tombol "Tarik Data" atau "Pull Data"
  7. Sistem menampilkan loading/progress bar
  8. Sistem melakukan koneksi ke mesin fingerprint (FTM)
  9. Sistem mengambil data absensi sesuai parameter yang ditentukan
  10. Sistem melakukan matching PIN dari FTM dengan NIK dari HRIS menggunakan tabel mapping
  11. Sistem memvalidasi kelengkapan data:
    • Cek apakah PIN ter-mapping ke NIK
    • Cek apakah karyawan masih aktif di HRIS
    • Cek duplikasi data (jika data sudah ada)
  12. Sistem menyimpan data yang valid ke database dengan flag "from_fingerprint"
  13. Sistem menampilkan summary hasil pull:
    • Total record yang berhasil di-pull
    • Jumlah data baru yang ditambahkan
    • Jumlah data duplikat yang di-skip
    • Jumlah data error/tidak valid (PIN tidak ter-mapping, dll)
  14. HRD dapat melihat detail log atau download report hasil pull
Skenario Alternatif A1: Koneksi ke Mesin Fingerprint Gagal
  1. Di langkah 8, jika sistem tidak dapat terhubung ke FTM
  2. Sistem menampilkan error message "Koneksi ke mesin fingerprint gagal"
  3. Sistem memberikan detail error (timeout, IP tidak dapat dijangkau, dll)
  4. HRD dapat:
    • Cek koneksi jaringan
    • Verify IP address dan port mesin
    • Retry koneksi
    • Contact IT support jika masalah berlanjut

A2: Ditemukan Data dengan PIN Tidak Ter-mapping
  1. Di langkah 11, jika ditemukan record dengan PIN yang belum di-mapping ke NIK
  2. Sistem tetap menyimpan data tersebut tapi dengan flag "unmapped"
  3. Sistem mencatat PIN yang unmapped dalam report
  4. Di summary, HRD dapat melihat daftar PIN yang perlu di-mapping
  5. HRD dapat langsung navigasi ke menu "Mapping PIN" untuk update

A3: Data Duplikat Terdeteksi
  1. Di langkah 11, jika sistem menemukan data yang sudah ada di database
  2. Sistem melakukan pengecekan berdasarkan: NIK/PIN + Tanggal + Waktu
  3. Jika benar-benar duplikat (data identik), sistem skip data tersebut
  4. Jika ada perbedaan waktu sedikit (selisih <1 menit), sistem ambil yang terbaru
  5. Sistem mencatat jumlah duplikat dalam summary

A4: Proses Pull Timeout atau Terlalu Lama
  1. Jika data yang di-pull sangat banyak (ribuan record) dan melebihi timeout
  2. Sistem memberikan warning "Proses memakan waktu lama, silakan tunggu"
  3. Sistem tetap lanjutkan proses di background
  4. Setelah selesai, sistem kirim notifikasi atau email ke HRD
  5. HRD dapat melihat hasil di menu "Log History Pull"

Deskripsi Use Case Transaksi - Verifikasi dan Editable Data Missing

ID Use Case UCT2
Nama Use Case Verifikasi dan Editable Data Missing
Tujuan Memungkinkan HRD untuk melakukan verifikasi terhadap data absensi yang tidak lengkap atau missing (tidak ada check-in/check-out), dan melakukan edit atau input manual untuk melengkapi data tersebut agar valid untuk pemrosesan payroll dan report.
Aktor Primer HRD
Aktor Sekunder a. HRIS (untuk validasi data karyawan)
b. Database Sistem Absensi
c. Sistem Notifikasi
Deskripsi Singkat Use case ini menjelaskan bagaimana HRD dapat mengidentifikasi data absensi yang tidak lengkap (missing check-in, missing check-out, atau tidak ada data sama sekali pada hari kerja), melakukan verifikasi dengan karyawan atau supervisor, dan mengedit data tersebut untuk melengkapi informasi absensi.
Precondition
  1. HRD telah login ke sistem
  2. Data absensi sudah di-pull dari fingerprint atau sudah ada di database
  3. Data jadwal kerja karyawan tersedia untuk identifikasi missing
  4. HRD memiliki hak akses untuk edit/verifikasi data
Postcondition
  1. Data missing teridentifikasi dan ditampilkan
  2. HRD berhasil melakukan edit/input untuk melengkapi data
  3. Data hasil edit tersimpan dengan flag "verified" atau "manual_input"
  4. Log aktivitas verifikasi tercatat (siapa, kapan, perubahan apa)
  5. Data siap untuk proses payroll atau report
Skenario Utama
  1. HRD login ke sistem
  2. HRD membuka menu "Transaksi > Data Missing" atau "Verifikasi Absensi"
  3. Sistem menampilkan form filter:
    • Periode tanggal
    • Departemen/Divisi
    • Tipe missing (No Check-in, No Check-out, No Data)
    • Status verifikasi (Belum/Sudah diverifikasi)
  4. HRD mengisi filter sesuai kebutuhan (misalnya: periode bulan ini, departemen IT)
  5. HRD klik "Cari" atau "Filter"
  6. Sistem melakukan analisa data absensi:
    • Compare data absensi dengan jadwal kerja karyawan
    • Identifikasi hari kerja yang tidak ada data check-in/out
    • Identifikasi data yang hanya ada check-in tanpa check-out atau sebaliknya
  7. Sistem menampilkan tabel data missing dengan kolom:
    • Tanggal
    • NIK & Nama Karyawan
    • Departemen
    • Jadwal (jam kerja seharusnya)
    • Check-in (ada/kosong)
    • Check-out (ada/kosong)
    • Jenis Missing (No In/No Out/No Data)
    • Status (Belum/Sudah Verified)
    • Action (Edit/Verifikasi)
  8. HRD dapat sorting atau export data untuk analisa
  9. HRD memilih salah satu record dan klik tombol "Edit" atau "Verifikasi"
  10. Sistem menampilkan form edit dengan field:
    • Tanggal (read-only)
    • Karyawan (read-only)
    • Waktu Check-in (editable - time picker)
    • Waktu Check-out (editable - time picker)
    • Status Kehadiran (dropdown: Hadir, Izin, Sakit, Cuti, Alpa, Dinas Luar)
    • Catatan/Alasan (required)
    • Upload Dokumen Pendukung (optional - foto surat, screenshot, dll)
  11. HRD mengisi atau melengkapi data yang missing
  12. HRD mengisi alasan verifikasi (contoh: "Sudah konfirmasi dengan supervisor", "Data dari email approval", dll)
  13. HRD bisa upload dokumen pendukung jika ada
  14. HRD klik tombol "Simpan" atau "Update"
  15. Sistem memvalidasi kelengkapan data (cek required fields)
  16. Sistem menyimpan data dengan flag:
    • "verified_by": ID user HRD
    • "verified_at": timestamp
    • "data_source": "manual_verification"
  17. Sistem mencatat log perubahan (before-after)
  18. Sistem update status record menjadi "Verified"
  19. Sistem menampilkan notifikasi "Data berhasil diupdate"
  20. HRD kembali ke list dan dapat melanjutkan verifikasi record lain
Skenario Alternatif A1: Bulk Verification (Verifikasi Massal)
  1. Di langkah 9, HRD dapat memilih multiple records dengan checkbox
  2. HRD klik "Bulk Edit" atau "Verifikasi Massal"
  3. Sistem menampilkan form untuk apply perubahan ke semua record terpilih
  4. HRD mengisi status dan alasan yang sama untuk semua record
  5. Sistem update semua record sekaligus dan log masing-masing perubahan

A2: Data Sudah Diverifikasi Sebelumnya
  1. Di langkah 10, jika HRD klik record yang sudah verified
  2. Sistem menampilkan warning "Data ini sudah diverifikasi sebelumnya"
  3. Sistem tampilkan info: siapa yang verify, kapan, dan alasan
  4. HRD dapat memilih:
    • "Edit Ulang" - untuk revisi verifikasi sebelumnya
    • "Cancel" - untuk kembali ke list
  5. Jika edit ulang, sistem simpan history verifikasi lama dan buat entry baru

A3: Validasi Gagal
  1. Di langkah 15, jika ada field required yang tidak diisi atau format waktu tidak valid
  2. Sistem menampilkan error message dengan detail field yang bermasalah
  3. HRD harus melengkapi atau memperbaiki data sebelum bisa save
  4. Form tetap terbuka dengan data yang sudah diisi (tidak hilang)

A4: Export Data Missing untuk Follow-up
  1. Di langkah 8, setelah data missing ditampilkan
  2. HRD klik tombol "Export to Excel"
  3. Sistem generate file Excel dengan list data missing
  4. HRD dapat share file ini ke supervisor untuk klarifikasi
  5. Setelah dapat info, HRD kembali ke sistem untuk verifikasi

A5: Reject Data (Tandai sebagai tidak valid)
  1. Di langkah 10, selain "Edit", HRD dapat klik "Reject" atau "Tandai Invalid"
  2. Sistem menampilkan dialog konfirmasi
  3. HRD mengisi alasan reject (contoh: "Karyawan keluar/resign", "Cuti panjang", dll)
  4. Sistem update status menjadi "Rejected/Invalid"
  5. Record ini tidak akan muncul lagi di list missing kecuali filter "Show Rejected"

Deskripsi Use Case Transaksi - Tambah Tukar Jadwal dan View History Tukar Jadwal

ID Use Case UCT3
Nama Use Case Tambah Tukar Jadwal dan View History Tukar Jadwal
Tujuan Memungkinkan HRD untuk mengelola permintaan pertukaran jadwal shift antar karyawan, melakukan approval atau rejection, serta melihat history semua transaksi tukar jadwal untuk keperluan audit dan monitoring.
Aktor Primer HRD
Aktor Sekunder a. HRIS (untuk data karyawan dan jadwal)
b. Database Sistem Absensi
c. Sistem Notifikasi (untuk notif ke karyawan)
d. Karyawan (indirect - pembuat request)
Deskripsi Singkat Use case ini menjelaskan bagaimana HRD dapat membuat permintaan tukar jadwal shift (atas nama karyawan atau inisiatif HRD), melakukan approval terhadap permintaan yang masuk, dan melihat history lengkap semua transaksi tukar jadwal yang pernah terjadi beserta status dan perubahannya.
Precondition
  1. HRD telah login ke sistem
  2. Data karyawan dan jadwal shift tersedia di HRIS
  3. Data mapping karyawan ke shift sudah dikonfigurasi
  4. HRD memiliki hak akses untuk create dan approve tukar jadwal
Postcondition
  1. Permintaan tukar jadwal tersimpan di database
  2. Jika approved, jadwal shift kedua karyawan ter-update otomatis
  3. Karyawan yang terlibat menerima notifikasi
  4. History tukar jadwal tercatat lengkap dengan status dan timeline
  5. Log aktivitas tersimpan untuk audit trail
Skenario Utama Bagian A: Tambah/Create Tukar Jadwal
  1. HRD login ke sistem
  2. HRD membuka menu "Transaksi > Tukar Jadwal"
  3. Sistem menampilkan halaman dengan 2 tab:
    • Tab "Daftar Permintaan" - untuk approval
    • Tab "Tambah Baru" - untuk create request
  4. HRD klik tab "Tambah Baru" atau tombol "+ Tambah Tukar Jadwal"
  5. Sistem menampilkan form dengan field:
    • Karyawan A (pencari dengan dropdown/autocomplete)
    • Jadwal Shift Karyawan A (auto-load setelah pilih karyawan)
    • Tanggal yang akan ditukar
    • Karyawan B (partner tukar - dropdown/autocomplete)
    • Jadwal Shift Karyawan B (auto-load)
    • Tanggal shift Karyawan B
    • Alasan Pertukaran (textarea - required)
    • Status Approval (dropdown: Pending, Approved, Rejected)
    • Catatan HRD (textarea - optional)
  6. HRD memilih Karyawan A (yang akan tukar shift)
  7. Sistem menampilkan jadwal shift Karyawan A untuk tanggal tertentu
  8. HRD memilih tanggal yang akan ditukar
  9. HRD memilih Karyawan B (partner tukar)
  10. Sistem validasi:
    • Apakah kedua karyawan dalam departemen/group yang sama (atau allowed)
    • Apakah shift kompatibel (tidak bentrok dengan aturan)
    • Apakah tidak ada cuti/izin pada tanggal tersebut
  11. HRD mengisi alasan pertukaran
  12. HRD memilih status:
    • Jika "Pending" - request akan masuk ke queue approval
    • Jika "Approved" - langsung approved dan shift berubah
    • Jika "Rejected" - request ditolak dengan alasan
  13. HRD klik tombol "Simpan"
  14. Sistem menyimpan permintaan tukar jadwal
  15. Jika status "Approved", sistem langsung update jadwal shift di database
  16. Sistem kirim notifikasi ke Karyawan A dan B
  17. Sistem menampilkan konfirmasi "Permintaan tukar jadwal berhasil disimpan"
  18. HRD kembali ke list permintaan

Bagian B: View History Tukar Jadwal
  1. Dari menu "Transaksi > Tukar Jadwal"
  2. HRD klik tab "History" atau "Riwayat"
  3. Sistem menampilkan filter history:
    • Periode (dari tanggal - sampai tanggal)
    • Karyawan (cari by nama/NIK)
    • Departemen
    • Status (All, Pending, Approved, Rejected)
  4. HRD mengisi filter atau langsung klik "Tampilkan History"
  5. Sistem menampilkan tabel history dengan kolom:
    • No
    • Tanggal Pengajuan
    • Karyawan A (NIK & Nama)
    • Shift Awal A → Shift Baru A
    • Karyawan B (NIK & Nama)
    • Shift Awal B → Shift Baru B
    • Tanggal Efektif Tukar
    • Status (badge: Pending/Approved/Rejected)
    • Approved/Rejected By
    • Tanggal Approval
    • Action (View Detail)
  6. HRD dapat sorting by tanggal atau status
  7. HRD dapat klik "View Detail" pada salah satu record
  8. Sistem menampilkan popup/modal detail history dengan informasi lengkap:
    • Info lengkap karyawan A dan B
    • Perubahan shift detail (before-after)
    • Alasan pertukaran
    • Timeline (diajukan kapan, approved kapan, dll)
    • Catatan HRD
    • Dokumen pendukung (jika ada)
  9. HRD dapat export history ke Excel/PDF untuk laporan
Skenario Alternatif A1: Approval Permintaan yang Pending
  1. Di Tab "Daftar Permintaan", HRD melihat list request dengan status Pending
  2. HRD klik "Review" atau "Approve/Reject" pada salah satu request
  3. Sistem menampilkan detail lengkap permintaan
  4. HRD review kelayakan pertukaran
  5. HRD memilih action:
    • "Approve" - setujui dan shift akan berubah otomatis
    • "Reject" - tolak dengan mengisi alasan penolakan
  6. Jika Approve: Sistem update status, update shift, kirim notif approval
  7. Jika Reject: Sistem update status, kirim notif rejection dengan alasan

A2: Validasi Konflik Shift Gagal
  1. Di langkah 10 (validasi), jika ditemukan konflik:
    • Karyawan sudah ada cuti/izin di tanggal tersebut
    • Shift tidak kompatibel (misal: shift malam ke shift pagi di hari yang sama)
    • Sudah ada request tukar jadwal lain yang bentrok
  2. Sistem menampilkan error message dengan detail konflik
  3. HRD harus memilih karyawan/tanggal lain atau cancel request

A3: Bulk Approval (untuk banyak request sekaligus)
  1. Di Tab "Daftar Permintaan", HRD dapat select multiple request dengan checkbox
  2. HRD klik "Bulk Approve" atau "Approve Selected"
  3. Sistem validasi semua request yang dipilih
  4. Sistem approve semua yang valid dan skip yang error
  5. Sistem tampilkan summary: berapa yang approved, berapa yang skip/error

A4: Cancel Tukar Jadwal yang Sudah Approved
  1. Di History, HRD menemukan tukar jadwal yang perlu di-cancel
  2. HRD klik "Cancel" atau "Batalkan"
  3. Sistem menampilkan konfirmasi "Apakah yakin membatalkan tukar jadwal ini?"
  4. HRD confirm dan isi alasan pembatalan
  5. Sistem rollback perubahan shift ke kondisi semula
  6. Sistem update status menjadi "Cancelled"
  7. Sistem kirim notifikasi pembatalan ke karyawan terkait

A5: Filter History dengan Multiple Criteria
  1. HRD dapat kombinasi filter (misal: Departemen IT + Status Approved + Periode Januari 2026)
  2. Sistem apply semua filter dan tampilkan hasil yang match semua criteria
  3. HRD dapat save filter sebagai "Quick Filter" untuk digunakan lagi

Deskripsi Use Case Transaksi - Kirim Surat Peringatan

ID Use Case UCT4
Nama Use Case Kirim Surat Peringatan
Tujuan Memungkinkan HRD untuk membuat, mengelola, dan mengirimkan surat peringatan (SP) kepada karyawan yang melakukan pelanggaran absensi atau kedisiplinan berdasarkan data kehadiran dan keterlambatan yang tercatat di sistem.
Aktor Primer HRD
Aktor Sekunder a. HRIS (untuk data karyawan dan email)
b. Database Sistem Absensi (untuk data pelanggaran)
c. Sistem Email/Notifikasi
d. Sistem Document Generator (untuk PDF SP)
e. Atasan/Supervisor Karyawan (indirect)
Deskripsi Singkat Use case ini menjelaskan bagaimana HRD dapat mengidentifikasi karyawan yang melakukan pelanggaran absensi (keterlambatan berulang, alpa, dll), membuat surat peringatan dengan template yang sudah tersedia, customize isi surat jika perlu, dan mengirimkannya via email atau print untuk diserahkan secara fisik kepada karyawan yang bersangkutan.
Precondition
  1. HRD telah login ke sistem
  2. Data pelanggaran absensi karyawan tersedia di database
  3. Template surat peringatan (SP 1, SP 2, SP 3) sudah dikonfigurasi
  4. Data email karyawan tersedia di HRIS
  5. HRD memiliki hak akses untuk create dan kirim SP
Postcondition
  1. Surat peringatan tersimpan di database dengan status "Draft" atau "Sent"
  2. Jika dikirim, karyawan menerima SP via email atau print
  3. Atasan/Supervisor menerima copy SP (cc)
  4. History SP karyawan ter-update (jumlah SP yang pernah diterima)
  5. Log aktivitas kirim SP tercatat untuk audit
  6. Dokumen PDF SP tersimpan di storage sistem
Skenario Utama
  1. HRD login ke sistem
  2. HRD membuka menu "Transaksi > Kirim Surat Peringatan" atau "Kirim SP"
  3. Sistem menampilkan dashboard dengan informasi:
    • Jumlah karyawan yang berpotensi dapat SP (berdasarkan threshold pelanggaran)
    • List karyawan dengan keterlambatan/alpa berulang
    • Quick stats: Total SP yang sudah dikirim bulan ini
  4. HRD dapat melihat list karyawan yang disarankan sistem untuk diberi SP
  5. HRD klik tombol "+ Buat Surat Peringatan" atau pilih karyawan dari list
  6. Sistem menampilkan form dengan field:
    • Cari Karyawan (autocomplete by NIK/Nama)
    • Tingkat SP (dropdown: SP 1, SP 2, SP 3)
    • Jenis Pelanggaran (dropdown: Keterlambatan, Alpa, Izin tidak resmi, dll)
    • Periode Pelanggaran (tanggal dari - sampai)
    • Detail Pelanggaran (auto-populated dari data absensi)
    • Tanggal Surat
    • Nomor Surat (auto-generate atau manual)
  7. HRD memilih karyawan yang akan diberi SP
  8. Sistem auto-load data karyawan dari HRIS:
    • Nama lengkap, NIK
    • Jabatan, Departemen
    • Email
    • Nama Atasan/Supervisor
  9. Sistem menampilkan history SP karyawan (jika sudah pernah dapat SP sebelumnya)
  10. HRD memilih tingkat SP (sistem bisa suggest berdasarkan history)
  11. HRD memilih jenis pelanggaran
  12. Sistem auto-generate detail pelanggaran dari data absensi:
    • Daftar tanggal terlambat dengan durasi
    • Daftar tanggal alpa
    • Summary: Total keterlambatan, total alpa, dll
  13. HRD dapat edit atau tambahkan detail pelanggaran manual jika perlu
  14. HRD klik "Preview Surat"
  15. Sistem generate preview surat berdasarkan template SP yang dipilih
  16. Preview menampilkan surat lengkap dengan:
    • Header perusahaan
    • Nomor dan tanggal surat
    • Perihal: Surat Peringatan [Tingkat]
    • Kepada: [Nama Karyawan]
    • Isi surat dengan detail pelanggaran
    • Sanksi atau tindakan yang akan diambil
    • Tanda tangan HRD Manager (digital)
  17. HRD review preview surat
  18. Jika perlu edit, HRD klik "Edit Content" untuk customize isi surat
  19. Setelah OK, HRD memilih metode pengiriman:
    • "Kirim via Email" - kirim soft copy ke email karyawan
    • "Generate PDF untuk Print" - download PDF untuk print dan serahkan fisik
    • "Keduanya" - kirim email dan generate PDF
  20. HRD klik "Kirim Surat Peringatan"
  21. Sistem meminta konfirmasi final: "Apakah yakin mengirim SP [Tingkat] ke [Nama Karyawan]?"
  22. HRD confirm
  23. Sistem proses:
    • Generate PDF surat peringatan
    • Simpan PDF ke storage/document management
    • Jika kirim email: Kirim email dengan attachment PDF ke karyawan
    • Kirim copy (cc) ke atasan/supervisor
    • Update history SP karyawan di database
    • Log aktivitas dengan detail timestamp, user, dll
  24. Sistem menampilkan notifikasi: "Surat Peringatan berhasil dikirim"
  25. HRD dapat:
    • Download PDF yang sudah di-generate
    • Lihat status email (terkirim/gagal)
    • Kembali ke dashboard untuk create SP lain
Skenario Alternatif A1: Email Karyawan Tidak Tersedia
  1. Di langkah 8, jika sistem tidak menemukan email karyawan di HRIS
  2. Sistem menampilkan warning "Email karyawan tidak tersedia"
  3. Opsi "Kirim via Email" menjadi disabled
  4. HRD hanya dapat pilih "Generate PDF untuk Print"
  5. HRD perlu update email di HRIS atau serahkan SP secara fisik

A2: Konfirmasi untuk SP Tingkat Tinggi (SP 3)
  1. Di langkah 22, jika tingkat SP adalah SP 3 (biasanya SP terakhir sebelum PHK)
  2. Sistem menampilkan konfirmasi extra dengan warning
  3. Sistem tampilkan info: "SP 3 adalah surat peringatan terakhir. Pelanggaran berikutnya dapat berakibat PHK"
  4. HRD harus confirm 2x atau input password ulang untuk security
  5. Sistem juga kirim notifikasi ke HR Manager atau Director untuk transparency

A3: Pengiriman Email Gagal
  1. Di langkah 24, jika pengiriman email gagal (mail server error, email invalid, dll)
  2. Sistem tetap simpan data SP dengan status "Email Failed"
  3. Sistem menampilkan error notification dengan detail error
  4. HRD dapat:
    • "Retry" - coba kirim ulang email
    • "Download PDF" - download untuk kirim manual via email lain
    • "Mark as Sent" - tandai sudah dikirim secara manual
  5. Log tetap mencatat attempt pengiriman dan error

A4: Bulk Create SP (untuk multiple karyawan sekaligus)
  1. Di langkah 5, HRD dapat pilih "Bulk Create SP"
  2. HRD select multiple karyawan dari list menggunakan checkbox
  3. HRD pilih tingkat SP dan jenis pelanggaran yang sama untuk semua
  4. Sistem generate preview untuk setiap karyawan (dengan data masing-masing)
  5. HRD review dan confirm
  6. Sistem proses semua SP sekaligus
  7. Sistem tampilkan summary: berapa SP berhasil dikirim, berapa yang gagal

A5: View History SP Karyawan
  1. HRD dapat klik "History SP" pada dashboard
  2. Sistem menampilkan filter: Karyawan, Periode, Tingkat SP, Status
  3. Sistem tampilkan tabel history semua SP yang pernah dikirim
  4. HRD dapat view detail SP, download PDF, atau resend jika perlu
  5. HRD dapat generate report history SP untuk periode tertentu

A6: Cancel/Recall SP yang Sudah Dikirim
  1. Jika ada kesalahan, HRD dapat cancel SP yang baru saja dikirim
  2. HRD pilih SP dari history dan klik "Cancel" atau "Recall"
  3. Sistem menampilkan konfirmasi dan meminta alasan pembatalan
  4. Sistem update status SP menjadi "Cancelled"
  5. Sistem kirim email pemberitahuan pembatalan ke karyawan (jika perlu)
  6. SP yang dibatalkan tetap tercatat di history dengan status cancelled

📌 Catatan Umum Use Case Transaksi

Perbedaan dengan Use Case Master Data:
Use case transaksi bersifat operasional dan mengubah state/status data di sistem. Setiap transaksi akan menghasilkan perubahan data dan harus di-log dengan baik untuk audit trail.

Integrasi Sistem:

  • FTM (Fingerspot): Source data absensi raw yang di-pull ke sistem
  • HRIS: Validasi data karyawan, email, struktur organisasi
  • Email Service: Untuk notifikasi dan pengiriman SP
  • Document Generator: Generate PDF surat peringatan dengan template

Best Practices:
  • Selalu log siapa yang melakukan transaksi, kapan, dan perubahan apa
  • Implementasi approval workflow untuk transaksi yang sensitive (tukar jadwal, SP tingkat 3)
  • Backup data sebelum bulk operation untuk recovery jika terjadi error
  • Notifikasi real-time ke pihak terkait agar transparency terjaga
