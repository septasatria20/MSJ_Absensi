Deskripsi Use Case Transaksi – Pull Data Finger

ID Use Case UCT1
Nama Use Case Pull Data Finger
Tujuan Memungkinkan admin untuk mengambil data absensi dari mesin fingerprint secara otomatis dan menyimpannya ke database sistem untuk keperluan monitoring kehadiran karyawan.
Aktor Primer Admin
Aktor Sekunder a. Mesin Fingerprint
b. Sistem Database Absensi
Deskripsi Singkat Use case ini menjelaskan proses saat admin melakukan pull/tarik data absensi dari mesin fingerprint. Sistem akan terhubung ke mesin fingerprint, mengambil data absensi dalam periode tertentu, memvalidasi data, dan menyimpannya ke database.
Precondition
  1. Admin telah login ke sistem
  2. Koneksi ke mesin fingerprint tersedia
  3. Admin memiliki akses ke menu Transaksi
Postcondition
  1. Data absensi dari fingerprint tersimpan di database
  2. Admin menerima notifikasi atau log hasil pull data
  3. Data dapat langsung dilihat di report kehadiran
Skenario Utama
  1. Admin mengakses menu "Transaksi > Pull Finger"
  2. Admin memilih mesin fingerprint yang akan di-pull
  3. Admin menentukan rentang tanggal data yang akan diambil
  4. Sistem melakukan koneksi ke mesin fingerprint
  5. Sistem mengambil data absensi sesuai parameter
  6. Sistem memvalidasi format dan kelengkapan data
  7. Sistem menyimpan data ke database dengan menghindari duplikasi
  8. Sistem menampilkan summary jumlah data yang berhasil di-pull
Skenario Alternatif A1: Koneksi ke mesin gagal
1. Di langkah 4, jika koneksi ke mesin fingerprint gagal
2. Sistem menampilkan pesan error "Koneksi ke mesin fingerprint gagal"
3. Admin dapat mencoba kembali atau membatalkan proses

A2: Data duplikat terdeteksi
1. Di langkah 7, jika ditemukan data yang sudah ada
2. Sistem akan skip data duplikat dan hanya menyimpan data baru
3. Sistem menampilkan informasi jumlah data duplikat yang di-skip

Deskripsi Use Case Transaksi – Input Manual Absensi

ID Use Case UCT2
Nama Use Case Input Manual Absensi
Tujuan Memungkinkan admin/HRD untuk menginput atau mengubah data absensi secara manual untuk mengatasi kondisi khusus seperti lupa absen, mesin error, atau karyawan yang sedang dinas luar.
Aktor Primer Admin, HRD
Aktor Sekunder Sistem Database Absensi
Deskripsi Singkat Use case ini menjelaskan proses saat admin atau HRD melakukan input absensi secara manual untuk karyawan. Sistem akan memvalidasi data yang diinput, menyimpan ke database, dan mencatat siapa yang melakukan input manual tersebut.
Precondition
  1. Admin/HRD telah login ke sistem
  2. Data karyawan sudah terdaftar di sistem
  3. Admin/HRD memiliki hak akses untuk input manual
Postcondition
  1. Data absensi manual tersimpan di database
  2. Sistem mencatat log input manual (user, waktu, alasan)
  3. Data absensi manual dapat dilihat di report dengan indikator "Manual"
Skenario Utama
  1. Admin/HRD mengakses menu "Transaksi > Input Manual"
  2. Admin/HRD mencari dan memilih karyawan
  3. Admin/HRD memilih tanggal absensi
  4. Admin/HRD mengisi waktu check-in dan check-out
  5. Admin/HRD memilih status kehadiran (Hadir, Izin, Sakit, dll)
  6. Admin/HRD mengisi alasan input manual (wajib)
  7. Admin/HRD dapat upload dokumen pendukung (opsional)
  8. Sistem memvalidasi kelengkapan data
  9. Admin/HRD menekan tombol "Simpan"
  10. Sistem menyimpan data dengan flag "manual_input" dan mencatat user yang input
  11. Sistem menampilkan pesan sukses
Skenario Alternatif A1: Data sudah ada
1. Di langkah 8, jika absensi untuk karyawan di tanggal tersebut sudah ada
2. Sistem menampilkan warning dan opsi untuk "Replace" atau "Cancel"
3. Jika "Replace", data lama akan diganti dengan data baru
4. Sistem menyimpan history perubahan

A2: Validasi gagal
1. Di langkah 8, jika ada field yang tidak valid atau kosong
2. Sistem menampilkan pesan error dengan detail field yang bermasalah
3. Admin/HRD harus memperbaiki data sebelum dapat menyimpan

Deskripsi Use Case Transaksi – Approval Tukar Jadwal

ID Use Case UCT3
Nama Use Case Approval Tukar Jadwal
Tujuan Memungkinkan admin/HRD untuk mereview dan menyetujui/menolak permintaan pertukaran jadwal shift antar karyawan untuk menjaga kelancaran operasional.
Aktor Primer Admin, HRD
Aktor Sekunder a. Karyawan (Pemohon)
b. Sistem Notifikasi
Deskripsi Singkat Use case ini menjelaskan proses approval permintaan tukar jadwal shift. Admin/HRD dapat melihat detail permintaan, memvalidasi ketersediaan, dan memberikan keputusan approval atau rejection dengan catatan.
Precondition
  1. Admin/HRD telah login ke sistem
  2. Ada permintaan tukar jadwal dengan status "Pending"
  3. Admin/HRD memiliki hak akses approval
Postcondition
  1. Status permintaan berubah menjadi "Approved" atau "Rejected"
  2. Karyawan yang terlibat menerima notifikasi hasil approval
  3. Jika approved, jadwal shift kedua karyawan akan berubah otomatis
  4. Log approval tersimpan di sistem
Skenario Utama
  1. Admin/HRD mengakses menu "Transaksi > Tukar Jadwal"
  2. Sistem menampilkan daftar permintaan tukar jadwal
  3. Admin/HRD memfilter atau mencari permintaan dengan status "Pending"
  4. Admin/HRD klik detail salah satu permintaan
  5. Sistem menampilkan informasi lengkap:
    • Data karyawan A (pemohon) dan karyawan B (partner)
    • Jadwal yang akan ditukar
    • Alasan pertukaran
    • Tanggal pengajuan
  6. Admin/HRD melakukan validasi ketersediaan dan kesesuaian
  7. Admin/HRD memutuskan untuk "Approve" atau "Reject"
  8. Admin/HRD mengisi catatan/alasan keputusan
  9. Sistem update status permintaan
  10. Jika "Approve", sistem update jadwal shift kedua karyawan
  11. Sistem mengirim notifikasi ke kedua karyawan
  12. Sistem menampilkan pesan konfirmasi
Skenario Alternatif A1: Catatan approval tidak diisi
1. Di langkah 8, jika admin/HRD tidak mengisi catatan saat reject
2. Sistem menampilkan warning bahwa catatan wajib untuk rejection
3. Admin/HRD harus mengisi catatan sebelum dapat submit

A2: Konflik jadwal terdeteksi
1. Di langkah 10, jika sistem mendeteksi konflik jadwal
2. Sistem menampilkan warning tentang konflik
3. Admin/HRD dapat memilih untuk tetap approve atau membatalkan

Deskripsi Use Case Transaksi – Cek Data Missing

ID Use Case UCT4
Nama Use Case Cek Data Missing
Tujuan Memungkinkan admin/HRD untuk mengidentifikasi dan menangani data absensi yang hilang atau tidak lengkap agar data kehadiran karyawan akurat dan lengkap.
Aktor Primer Admin, HRD
Aktor Sekunder Sistem Database Absensi
Deskripsi Singkat Use case ini menjelaskan proses identifikasi data absensi yang missing (tidak ada check-in/out, data tidak complete, dll). Admin dapat melihat daftar anomali dan melakukan tindakan korektif.
Precondition
  1. Admin/HRD telah login ke sistem
  2. Data absensi tersedia untuk dianalisa
  3. Sistem memiliki rules untuk mendeteksi data missing
Postcondition
  1. Daftar data missing teridentifikasi
  2. Admin/HRD dapat melakukan follow-up atau input manual
  3. Report data missing dapat diexport untuk keperluan analisa
Skenario Utama
  1. Admin/HRD mengakses menu "Transaksi > Data Missing"
  2. Sistem menampilkan form filter dengan opsi:
    • Periode tanggal
    • Department/Group karyawan
    • Jenis anomali (tidak ada check-in, tidak ada check-out, dll)
  3. Admin/HRD mengisi filter dan klik "Cari"
  4. Sistem melakukan analisa data absensi
  5. Sistem menampilkan tabel data missing dengan informasi:
    • NIK dan Nama Karyawan
    • Tanggal
    • Jenis anomali
    • Status penanganan
    • Action button
  6. Admin/HRD dapat melakukan action:
    • Input manual untuk melengkapi data
    • Tandai sebagai "Tidak perlu ditindaklanjuti" dengan alasan
    • Export ke Excel untuk follow-up
  7. Sistem mencatat tindakan yang dilakukan
  8. Sistem update status data missing
Skenario Alternatif A1: Tidak ada data missing
1. Di langkah 5, jika tidak ditemukan data missing
2. Sistem menampilkan pesan "Tidak ada data missing dalam periode tersebut"
3. Admin/HRD dapat mengubah filter atau keluar dari halaman

A2: Banyak data missing
1. Di langkah 5, jika ditemukan banyak data missing (>100 record)
2. Sistem menampilkan warning dan saran untuk mempersempit filter
3. Admin dapat melanjutkan melihat semua atau menyesuaikan filter

Deskripsi Use Case Transaksi – Kirim Surat Peringatan

ID Use Case UCT5
Nama Use Case Kirim Surat Peringatan
Tujuan Memungkinkan HRD untuk membuat dan mengirimkan surat peringatan kepada karyawan yang melakukan pelanggaran absensi secara otomatis melalui sistem.
Aktor Primer HRD
Aktor Sekunder a. Karyawan (Penerima SP)
b. Sistem Email/Notifikasi
c. Atasan/Supervisor Karyawan
Deskripsi Singkat Use case ini menjelaskan proses pembuatan dan pengiriman surat peringatan berdasarkan data pelanggaran absensi. HRD dapat memilih karyawan, jenis pelanggaran, tingkat SP, dan mengirimnya secara digital.
Precondition
  1. HRD telah login ke sistem
  2. Data pelanggaran absensi karyawan tersedia
  3. Template surat peringatan sudah dikonfigurasi
  4. Email/kontak karyawan tersedia di sistem
Postcondition
  1. Surat peringatan tercatat di database
  2. Karyawan menerima surat peringatan via email/notifikasi
  3. Atasan/Supervisor menerima copy surat peringatan
  4. History SP karyawan terupdate
Skenario Utama
  1. HRD mengakses menu "Transaksi > Kirim SP"
  2. Sistem menampilkan daftar karyawan yang memiliki pelanggaran
  3. HRD dapat melihat detail pelanggaran (jenis, frekuensi, tanggal)
  4. HRD memilih karyawan yang akan diberi SP
  5. HRD memilih tingkat SP (SP 1, SP 2, SP 3)
  6. Sistem menampilkan preview surat dengan template yang sesuai
  7. HRD dapat mengedit konten surat jika diperlukan
  8. HRD memilih metode pengiriman (Email, Print, atau keduanya)
  9. HRD memasukkan nama saksi/pihak terkait (jika ada)
  10. HRD klik tombol "Kirim Surat Peringatan"
  11. Sistem menyimpan data SP ke database
  12. Sistem mengirim email ke karyawan dan atasan
  13. Jika dipilih print, sistem generate PDF untuk dicetak
  14. Sistem menampilkan konfirmasi pengiriman sukses
Skenario Alternatif A1: Email karyawan tidak tersedia
1. Di langkah 12, jika email karyawan tidak ada atau tidak valid
2. Sistem menampilkan warning
3. HRD hanya dapat memilih opsi "Print" atau perlu update email terlebih dahulu

A2: Konfirmasi untuk SP tingkat tinggi
1. Di langkah 10, jika tingkat SP adalah SP 3 (terakhir)
2. Sistem menampilkan dialog konfirmasi tambahan
3. HRD harus confirm ulang sebelum SP dikirim

A3: Pengiriman email gagal
1. Di langkah 12, jika pengiriman email gagal
2. Sistem tetap menyimpan data SP dengan status "Email Failed"
3. Sistem menampilkan notifikasi error
4. HRD dapat retry pengiriman atau menggunakan metode lain
